# Phases of the Workflow

Detailed description of each phase, its inputs, outputs, and quality gates.

---

## Phase 1: Planning

**Owner**: `planner` subskill

**Input**:
- User request message
- Task ID (generated by orchestrator)

**Output**:
- `plan_<taskId>.md` in the project workspace
- Updated task manifest with `plan` artifact path
- Status set to `planning_done` (or `designing` if design phase is combined)

**Contents of Plan**:
- Objective statement
- Technology stack choices with justification
- Phased subtask breakdown (with IDs, titles, estimated hours)
- Dependencies between subtasks
- Acceptance criteria
- Risk assessment and mitigations
- Optional: initial architecture sketch

**Quality Gate**:
- Plan must include at least 5 subtasks for any non-trivial project.
- Each subtask must have clear acceptance criteria.
- Tech stack must match typical conventions (e.g., use well-known libraries).
- No open dependencies that block start (e.g., "wait for API credentials" unless unavoidable and noted).
- Review by orchestrator or user (optional).

---

## Phase 2: Design

**Owner**: `designer` subskill (optional; can be merged with planning)

**Input**:
- The plan document

**Output**:
- `DESIGN.md`
- Schema files (if database)
- API specification (OpenAPI if possible)

**Contents**:
- Architecture diagram (ASCII or description)
- Component responsibilities
- Data model (SQL or NoSQL schema)
- REST/GraphQL API endpoints with request/response formats
- Security considerations (auth, input validation, CORS)
- Deployment topology
- Environment variables list
- Error handling strategy

**Quality Gate**:
- All CRUD operations covered by API design.
- Database schema normalized appropriately.
- Security measures documented.
- No external service dependencies without mock strategy.
- Review score >= 8.0 (if separate review phase).

---

## Phase 3: Implementation

**Owner**: `coder` subskill

**Input**:
- Design document (or plan if design omitted)
- Task manifest

**Output**:
- Full source code in workspace
- `package.json` / `requirements.txt` / `Cargo.toml` etc.
- `.gitignore` present
- Git repository initialized with logical commits
- Server starts without errors
- Tests can run (possibly implemented by tester later)

**Standards**:
- Follow language/framework conventions.
- Use parameterized queries for databases.
- Escape user output to prevent XSS.
- Handle errors with appropriate HTTP status codes.
- Include inline comments on complex logic.
- Do not commit secrets; use environment variables.

**Quality Gate**:
- `npm start` (or equivalent) launches server without crashing.
- All endpoints from design are implemented.
- No lint errors (if lint tool exists).
- Git history present with at least one commit per subtask.
- Readme with quick start instructions (can be refined later).

---

## Phase 4: Testing

**Owner**: `tester` subskill

**Input**:
- Running server (usually launched by coder or orchestrator)
- API specification

**Output**:
- `test/test.js` (or equivalent)
- `artifacts/<taskId>/test_report.txt`
- Updated task manifest with `testing` status and results

**Test Coverage**:
- For each API endpoint: at least one happy path test.
- For create/update: validate required fields, error on invalid.
- For delete: verify removal and 404 on subsequent fetch.
- For edge cases: extremely long inputs, special characters, nulls.

**Quality Gate**:
- All tests pass (exit code 0).
- Test report shows >= 90% of critical paths covered.
- No flaky tests (order independent).
- If tests fail, task goes back to coder with failure details.

---

## Phase 5: Review

**Owner**: `reviewer` subskill

**Input**:
- Source code
- Test report
- Design document

**Output**:
- `artifacts/<taskId>/review_notes.md`
- Task manifest updated with `review` score and status

**Rubric**:
- Code Quality (0-10): style, modularity, duplication
- Security (0-10): SQL injection, XSS, CORS, secrets
- Maintainability (0-10): naming, structure, comments
- Performance (0-10): efficient queries, no N+1, proper indexing
- Testing (0-10): coverage, quality, edge cases
- Documentation (0-10): README, inline, design doc

**Quality Gate**:
- Overall score >= 8.0 to proceed.
- Any critical security issue (score < 5 in Security) blocks deployment regardless of total.
- Review notes must include at least one actionable suggestion if score < 9.0.
- If rejected, return to coder with review notes.

---

## Phase 6: Deployment

**Owner**: `deployer` subskill

**Input**:
- Reviewed, approved codebase
- Review report

**Output**:
- Deployment scripts (`scripts/setup-tunnel.sh`, `scripts/start-tunnel.sh`)
- Tunnel configuration template (`config/tunnel-config.yml`)
- Possibly systemd service file (`config/cloudflared-todo.service`)
- `DEPLOYMENT.md` with step-by-step instructions
- Publicly accessible URL (if tunnel created)
- Artifact paths in manifest; `status` becomes `deploying` then `deployed`

**Process**:
- Ensure server runs locally (`npm start`).
- Check `cloudflared` is installed.
- If user wants a temporary tunnel, run `cloudflared tunnel --url http://localhost:3000` and capture URL.
- If user wants persistent tunnel, they must run `setup-tunnel.sh` with their domain and credentials; the orchestrator may generate the config but cannot complete DNS changes without user's Cloudflare credentials.
- Provide instructions for systemd setup, log viewing, restart.

**Quality Gate**:
- Public URL responds with the application UI (HTTP 200).
- Health check (if exists) returns 200.
- Cloudflare Tunnel connection established (no 502).

---

## Phase 7: Documentation

**Owner**: `documenter` subskill

**Input**:
- All previous artifacts (plan, design, code, tests, review, deployment)

**Output**:
- Comprehensive `README.md` (features, quick start, API table, project structure, troubleshooting)
- `USER_GUIDE.md` (tutorials, UI walkthrough, FAQ)
- Possibly `CHANGELOG.md` (if versioned)
- Update task manifest `status` to `documenting` then `done`

**Standards**:
- README must include:
  - Badges (build status, version)
  - Feature list
  - Quick start (local run in < 5 steps)
  - API documentation table (method, endpoint, purpose, request/response example)
  - Testing instructions
  - Deployment options
  - Environment variables table
  - Contributing guidelines (optional)
  - License (if any)
- User Guide must be friendly, include screenshots (optional), and answer common questions.

**Quality Gate**:
- README provides enough info for a new developer to clone and run the project.
- All major features documented.
- Deployment guide matches actual scripts.
- No placeholder text (e.g., "TODO" entries) remaining.

---

## Final Status

When documentation is complete, the orchestrator sets:

```json
{
  "status": "done",
  "progress": 100,
  "completedAt": "<now>"
}
```

and notifies the user.

---

## Phase Skipping Rules

- If a phase is not required (e.g., design for a trivial script), the orchestrator may skip it but must record a note in the manifest.
- Minimum required phases: Planning, Implementation, Testing, Documentation.
- Deployment and Review are strongly recommended for production projects.

---

## Revisiting Phases

If any gate fails, the orchestrator should:

1. Set `status` to the failing phase name (e.g., `testing_failed`).
2. Add a `notes` field explaining the failure.
3. Notify the user and/or re-queue the responsible subskill with feedback.

Example: If tests fail, send back to coder: "tests failing, here is error log".

## Model Selection Strategy (Optional)

You can assign different LLMs to different phases via the `phaseModels` field in the task manifest. This allows cost optimization without sacrificing quality where it matters most.

| Phase | Recommended Model | Rationale |
|-------|------------------|-----------|
| Planning | GPT-4 Turbo / Claude 3 Opus | Requires deep reasoning and structured output |
| Design | GPT-4 Turbo / Claude 3 Opus | Complex architecture decisions |
| Implementation | GPT-4o-mini / Claude 3.5 Sonnet | High output volume; cost-sensitive |
| Testing | GPT-4o-mini / Claude 3 Haiku | Test code is repetitive and straightforward |
| Review | GPT-4 Turbo / Claude 3 Opus | Security and quality analysis benefit from top-tier reasoning |
| Deployment | GPT-4o-mini / Claude 3 Haiku | Scripts generation; low complexity |
| Documentation | GPT-4o-mini / Claude 3 Haiku | Document drafting; high volume |

Set these in your task manifest before starting the workflow. The orchestrator will pass the model to each `sessions_spawn` call, overriding the agent's default.
